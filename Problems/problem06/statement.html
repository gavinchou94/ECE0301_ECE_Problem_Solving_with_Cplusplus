Problem Statements: 

In this problem, you will write functions utilizing pointers and dynamic memory management to interact with a dynamically allocated array representing a database (the database for the solar system that is obtained from the NASA Horizons system). Follow the instructions below to implement functions for interacting with the database, write your own tests for the functions, and submit your code via Gradescope to check your solution against the autograder tests and memory checking. Action items are shaded in *** yellow.

  1. Download the starter code open it in VS Code and run until CMake: Configure.


  2. Open the provided files database.hpp and problem06_tests.cpp. Follow the remaining steps to implement the required functions in database.hpp, and add tests in problem06_tests.cpp. It is strongly recommended that you implement the functions one at a time, testing as you go, and submitting to Gradescope occasionally, which will be using Valgrind to check for memory leaks and errors.

    About Valgrind: The autograder for this problem will first run tests similar to the previous problem and output the number of tests passed and failed. Then, it will re-run the tests with the memory-checking tool Valgrind. Running Valgrind creates an output that lists the name of every test case, even if it passes, interspersed with any memory errors that are detected. The very end of the autograder output will summarize any memory leaks and errors. If memory is noted as in use at exit, then that is a memory leak (memory was allocated somewhere and not freed). If a memory error is noted but no leaks, then it could be bad access (out of bounds, for example). Errors will be listed in the test output in the order that they occur, so you should scroll through the test cases to find the descriptions of any memory errors. *** Part of your manual grading would be based on memory leak detection.

  3. More about the database: it holds information about solar system objects, with the following fixed fields that represent columns in the database: ID, Classification, Mean radius, Mass, Density, Orbital period
    1. Each row in the database will represent an “entry” with information about one solar system object.
    2. Each object will have a unique numerical identifier (ID) defined as follows (using the NASA JPL IDs for coordinate centers):
      Sun: 10; Mercury: 199; Venus: 299; Earth: 399; Earth’s Moon: 301; Mars: 499
    3. Each object will have a numerical Classification of the star (0), planet (1), dwarf planet (2), planetary satellite (3)
    4. The database itself will be a dynamic array of type double, which we will treat as a 2D array with a fixed number of columns (fields) and a variable number of rows (database entries).
    5. Initial data for the database is provided in constants.hpp as the 2D array DATA. Do not modify the provided constants.
    6. The database fields (column names) are constant and are therefore stored in a constant array of std::string provided in constants.hpp. This will be used as a parallel array to map the column names to column indices in our database.

  4. Read the comments for the provided functions and the provided test case. Make sure that you understand them before continuing. *** Finish the database function implementation in the database.hpp according to the specifications below.
    1. search: Search the specified field column in the database for a specified value; return true if found, and false if not. Note that the database values are floating point values, so any checks for equality should be approximate, using the threshold defined in constants.hpp.
    2. add_entry: Increase the size of the database and add a new entry (a new row of data). Note that both the database pointer and the rows arguments are passed by reference in the function definition (i.e., double *& database, int& rows) – you should update the pointer to point to the new resized database containing the new entry, and also update the count of rows accordingly. Use the Lecture 5: pointers in-class example function append as a guide, read the comments given in pointers_6.cpp in pointers_steps/ folder, but in this case, when you reallocate, you need to increase the array size by an entire row: database = new double[(rows+1)*COLS]. The database ID must be unique, therefore you should check for duplicates before adding a new entry (Hint: remember that your previous function search can help you check for duplicates). Return true if the add was successful, or false otherwise.
    3. remove_entry: Remove the row of data with the specified ID, and resize the database accordingly. Return true if the entry is found and removed, and false if the entry with the specified ID is not found. (Hint: again, remember that your previous function search can help you determine whether you can remove)

  5. *** Add test cases in problem06_tests.cpp to include at least 5 of the following categories. You are free to add more corner cases:
    • Test search for missing values and fields: for example, search for a non-existing field "Temperature" and search for a non-existing value at a given field "Class" = 3.1415
    • Test add_entry with a new row and search accordingly: for example, the entry for the moon looks like {301, 3, 1737.4, 7.349e22, 3.3437, 27.321582}
    • Test add_entry with a duplicated row and search accordingly: for example, add the entry for earth again, test that it should return false and database content should be unchanged
    • Test remove_entry with an existing ID and search accordingly: for example, remove the entry for the sun, and search "Class"=0 afterward.
    • Test add first and remove later, search in-between and afterward.
